# 要件定義書：番頭さん

**一行コンセプト：** 伝えたいことを話すだけで、マンション中に届く。

---

## 1. プロダクト概要

「番頭さん」は、マンション管理組合の高齢理事を主な対象に、「AIを使っている」という意識を一切持たせずに、議事録・お知らせ・デジタルサイネージへのコンテンツ配信を「話しかけるだけ」で完結させるWebサービスです。

既存のマンションDXサービスの多くは、管理会社やITリテラシーの高いユーザーを前提とした設計になっており、高齢者が多数を占める実際の管理組合の現場とのミスマッチが生じています。デジタルサイネージに至っては、コンテンツ更新が面倒で放置されるケース・専任担当者がいなくなって廃止になるケースが後を絶ちません。「番頭さん」は、昔の旅館や商家で主人の代わりに何でも取り仕切ってくれた「番頭さん」のように、難しい操作を一切見せずに、やりたいことを実現してくれる存在を目指します。

本プロダクトは、将来的に管理組合向けAI基盤サービスへと発展させることを見据えており、第一フェーズとして「はり出し（サイネージ出力スキル）」の実装からスタートします。まず「メモを書くだけでお知らせコンテンツができる」「できた内容を掲示板やサイネージに貼り出せる」という体験を小さく証明し、全体基盤への説得材料を積み上げます。

**コアコンセプト：「番頭さんに頼むだけで、マンションが動く」**

---

## 2. なぜこのアプローチを取るのか：戦略的背景

### サイネージ単体では戦えない

デジタルサイネージのコンテンツ生成・配信を軸にしたサービスは、すでに大手（ドコモ、シャープ等）から中小向けSaaSまで多数存在する。この市場にいきなり正面から参入しようとすると、機能・価格・販売網のいずれでも既存プレイヤーに対抗するのは難しい。そのため、まずターゲットを大きく絞り込み、既存サービスが取りこぼしている特定の層に特化したプロダクトを作ることを選んだ。

### 「ペインはあるが、お金が動いていない層」を狙う

ビジネスとして成立させるには、「困っているが、まだ誰もお金を払っていない」という未開拓の課題を見つける必要がある。オフィスビル・商業施設・ホテルといった用途にもサイネージの需要はあるが、これらはすでに競合が存在し、担当者のITリテラシーも高い。一方、マンション管理組合の高齢理事という層は、明確なペインを抱えながらも既存のDXサービスを使いこなせておらず、ニーズに対して解決策がまだ届いていない。

### 行政も問題視する「担い手不足」という構造的課題

この層を狙う根拠は、感覚的なものではなく社会的な裏付けがある。

国土交通省の調査によると、管理組合の役員を引き受けない理由として「**高齢化**」や「**何をしたらよいかわからない**」という回答が最も多く挙げられており、国交省・法務省は2025年3月、区分所有法・マンション管理適正化法等の改正案を閣議決定・国会提出している（2026年4月1日施行予定）。改正の背景には、築40年以上のマンション増加・所有者の高齢化・管理組合の担い手不足という「**3つの老い**」の深刻化がある。

国土交通省「令和5年度マンション総合調査」（2024年6月公表）では、昭和59年以前の築古マンションにおいて**70歳以上の区分所有者が55.9%を超える**ことが示されており、建物の老朽化と住民の高齢化が同時進行する「二つの老い」は、今後さらに深刻化する見通しだ（2034年末には築40年以上のマンションが現在の約2倍・293万戸に増加すると予測）。

また、国土交通省の別調査では、役員就任を要請された場合に「引き受けない」理由として**「高齢化のため」が39.9%**で最多であり（平成30年度マンション総合調査）、次世代への引き継ぎができないまま管理組合運営が停滞するケースが全国で増加している。

### 「AIが使えない世代」にこそ、AIライクな体験を届ける意義

サイネージ・コンテンツ生成・配信といったAIの恩恵を最も受けにくい世代が、実は最もその恩恵を必要としている。本プロダクトで「AIという言葉を使わずに、AIの力を届ける」インターフェースを作ることは、単なる高齢者向けサービスの開発にとどまらない。**「操作しない・意識しない・でも動いている」というUI設計の原型**を、最も難易度の高いユーザー層で実証することになる。これは将来、オフィスビル・商業施設・医療・自治体など他の領域にサービスを展開していくうえでも、強力な差別化の基盤となる。

### まず管理組合×サイネージで原型を作り、広げる

今回はあえて「最もAIを使いこなせない世代・最もサイネージコンテンツの必要性と意義が高い顧客層」として管理組合に絞り、そこで動くプロダクトを作る。この原型が成立すれば、同じ設計思想をより広い市場へと展開できる。**「番頭さん」は、そのための最初の一手である。**

## 3. 対象ユーザーと課題

### ターゲットユーザー像

東京都内を中心とした中小規模マンション（50〜200戸）の管理組合理事長・理事。年齢層は60〜75歳が中心で、スマートフォンは使えるがアプリの新規設定や複雑な操作には強い抵抗感がある。毎年の輪番制で理事を担当しており、ITの専門知識はなく、「誰かに頼みたいけど頼める人がいない」という状況に置かれている。管理会社はいるものの、細かいお知らせ作成や掲示対応は組合側が担うケースが多い。

### 課題①：議事録作成が重すぎる

理事会・総会の後、議事録を手書きやWordで作成するのに数時間かかる。書記担当の理事に負担が集中し、「議事録を作るのが嫌で理事を断る」ケースもある。区分所有法第42条により保管義務があるため、省略もできない。

### 課題②：お知らせ作成・配信に手間がかかる

回覧板・掲示板へのお知らせ作成は、文章を考えてWordで整形して印刷して貼り出すという多段階のプロセスが必要。工事・点検・総会案内など種類も多く、同じ情報を媒体ごとに作り直す二度手間が常態化している。

### 課題③：デジタルサイネージが活用されていない

エントランスや掲示板にサイネージを設置しているマンションでも、コンテンツ更新が面倒で同じ画面が何ヶ月も放置されているケースが多い。専用ソフトの操作が難しく更新担当者が固定化・属人化しており、担当理事が交代すると誰も操作できなくなる。

### 課題④：情報が古くなって廃止サイクルに陥る

コンテンツを一度作ったあと更新が止まり、古い情報がループ表示されることで逆効果になる。最終的にサイネージ自体を撤去する事態に追い込まれるケースも珍しくない。

### 課題⑤：既存DXサービスが使えない

Mcloud・osumaitなどの既存サービスは機能が豊富だが、操作が複雑でITリテラシーの低い高齢理事には使いこなせない。「管理会社が使うもの」という印象が強く、組合が直接使うイメージが持てない。

## 4. システム機能要件

### 機能①：はり出し（サイネージ出力スキル）

本プロダクトのPhase 1コア機能。管理組合に関連する情報（メモ書き・お知らせ内容・議事録テキスト）を入力すると、デジタルサイネージ・掲示板・回覧板向けのコンテンツを自動生成する。UIにはAIという言葉を一切使わず、「番頭さんが考えています…」という表現でローディングを演出する。

- **テキスト入力→サイネージコンテンツ生成**：「今月の理事会でエントランスの電球交換が決まりました」のようなメモ書きを入力するだけで、大きな文字・シンプルなレイアウトのサイネージ用コンテンツを生成する。
- **文体・フォント自動調整**：高齢居住者にも読みやすい平易な日本語・大きめフォント想定のテキストに自動変換する。難しい漢字には読み仮名を自動付与するオプションも持つ。
- **サイネージ用画像の自動生成**：テキストだけでなく、お知らせの内容に合ったビジュアル（季節感のあるイラスト・アイコン等）を組み合わせたサイネージ用画像を自動生成する。
- **ワンソース・マルチアウトプット**：1回の入力から、サイネージ用画像・掲示板用PDF・回覧板用テキストを同時生成し、用途別に出力フォーマットを切り替えられる。
- **プレビュー確認**：生成されたコンテンツを画面上でプレビューし、「これでいいですか？」の確認を経てから配信・印刷できる。

### 機能②：コンテンツ鮮度の自動管理（はり出し付随機能）

一度貼り出したコンテンツが古くなっても気づけない問題を解消する機能。

- **賞味期限アラート**：一定期間表示し続けたコンテンツは「そろそろ新しいお知らせはいかがですか？」と番頭さんが提案する。
- **季節・行事に応じた提案**：「もうすぐ大規模修繕の時期では？」「防災訓練のお知らせを作りませんか？」のように、時期に合った掲示内容を番頭さんが先回りして提案する。

### 機能③：議事録まとめ（将来スキル・Phase 2）

理事会・総会の音声録音データをアップロードするだけで、文字起こし・要約・議事録の下書きを自動生成する。

- **音声→テキスト変換**：録音データをアップロードすると自動で文字起こしを行う。
- **議事録フォーマット生成**：決定事項・アクションリスト・次回議題候補を自動で整理する。法的要件（区分所有法第42条）に対応した書式で出力する。
- **はり出し連携**：生成した議事録要約を「はり出し」にワンタップで送り、サイネージ・掲示板配信まで完結させる。

### 機能④：おしらせ配信（将来スキル・Phase 3）

生成したコンテンツを、サイネージ・掲示板・居住者アプリ・メールへ一括配信する。

- **配信先管理**：サイネージ端末・掲示板・メール・LINEなど配信先をまとめて管理する。
- **配信スケジュール設定**：「来週月曜から2週間掲示」のような自然言語での指定ができる。
- **既読・確認状況の把握**：配信したお知らせの閲覧状況を把握できる。

### 機能⑤：緊急・防災モード（将来スキル・Phase 3）

地震・台風・停電・近隣事故など緊急時に、サイネージ表示を瞬時に切り替える機能。

- **緊急ワンタップ切り替え**：管理画面からワンタップするだけで「緊急モード」へ移行し、通常コンテンツを強制中断して緊急情報画面に差し替える。
- **緊急用サイネージ画面の自動生成**：緊急種別（地震・停電・火災・大雨など）を選択すると、避難経路・緊急連絡先・状況説明を組み込んだ画面を自動生成する。
- **居住者向け緊急一斉連絡**：サイネージの切り替えと同時に、登録済みの居住者宛てに緊急メール・LINEを一斉送信する。

## 5. 画面構成案

### 画面①：ホーム（トップ）

「今日は何をお手伝いしましょうか？」という番頭さんの一言とともに、大きなボタンで主要機能に誘導する画面。難しいメニュー階層は持たず、「お知らせを作る」「議事録をまとめる」「貼り出す」の3択から始まる。文字は大きく（最小18px）、色はシンプルに。コンテンツ鮮度アラートがある場合はここに表示する。

### 画面②：お知らせ作成画面

伝えたい内容をテキストボックスに入力する画面。「どんなことをお知らせしたいですか？メモ書きで大丈夫です」という問いかけ形式にし、入力欄は大きく1つだけ。テンプレート（工事のお知らせ・設備点検・総会案内・防災訓練など）から選ぶこともできる。入力後「番頭さんに作ってもらう」ボタンを押すと生成が始まる。

### 画面③：プレビュー・確認画面

生成されたコンテンツのプレビューを表示する画面。サイネージ用・掲示板PDF・回覧板テキストをタブで切り替えて確認できる。「これでいいですか？」という確認メッセージとともに、「貼り出す」「直してもらう」「やり直す」の3択ボタンを表示。操作の迷いが生じないよう選択肢を最小限に絞る。

### 画面④：貼り出し完了画面

「貼り出しました！」という完了メッセージと、配信先・配信日時を表示する画面。シンプルな完了画面にし、次のアクション（ホームに戻る）を大きなボタンで案内する。

### 画面⑤：過去のお知らせ一覧

過去に作成・配信したお知らせの一覧を表示する画面。理事が交代しても引き継ぎができるよう履歴として保管される。コンテンツの賞味期限ステータスもここで一覧確認できる。

### 画面⑥：管理者設定画面（Phase 3以降）

配信するサイネージ画面の登録・グループ管理、居住者のメール・LINE配信リスト管理を行う画面。初期設定は管理士や管理会社がサポートすることを想定し、理事自身が触る頻度は最小限に設計する。

## 6. 技術スタック・アーキテクチャ方針

| 領域 | 候補・方針 |
|------|-----------|
| フロントエンド | Next.js（App Router）/ TypeScript / Tailwind CSS |
| バックエンド・DB | Supabase（認証・データ保管・RLS） |
| 認証 | Supabase Auth（Google OAuth対応） |
| AI文章生成エンジン | Claude API（コンテンツ生成・文体変換・議事録要約） |
| AI画像生成 | DALL-E 3 / Stable Diffusion API（サイネージ画像生成・Phase 1後半〜） |
| 音声文字起こし | OpenAI Whisper API（Phase 2以降） |
| サイネージ出力 | HTML生成→PDF変換 / Googleスライド連携（候補） |
| ホスティング | Vercel |
| 開発手法 | Vibe Coding（Claude Code使用） |

AIという言葉をUIに出さないため、Claude APIはバックエンドに完全に隠蔽し、ユーザーには「番頭さんが考えています…」のような表現でローディングを演出する。画像生成APIも同様に「番頭さんがお知らせを仕上げています…」と統一した表現で包む。

## 7. 開発フェーズ計画

### Phase 1（MVP）：はり出しスキルの実証

**ゴール：** テキストを入力するとサイネージ用コンテンツが生成・プレビューでき、PDFとして印刷できる状態を作る。

- [ ] プロジェクト初期設定（Next.js / Supabase / Claude API連携）
- [ ] お知らせ作成画面の実装（テキスト入力・テンプレート選択）
- [ ] Claude APIによるコンテンツ生成機能の実装
- [ ] サイネージ用・掲示板PDF・回覧板テキストのプレビュー画面実装
- [ ] 過去のお知らせ一覧画面の実装
- [ ] 高齢者向けUI調整（文字サイズ・ボタンサイズ・配色・コントラスト）
- [ ] Vercelへのデプロイ

### Phase 2：議事録まとめスキルの追加

**ゴール：** 音声録音をアップロードすると議事録の下書きが生成され、そのままはり出しに連携できる状態を作る。

- [ ] 音声ファイルアップロード機能の実装
- [ ] Whisper APIによる文字起こし機能の実装
- [ ] 議事録フォーマット自動生成機能の実装（区分所有法対応書式）
- [ ] はり出しへのワンタップ連携機能の実装
- [ ] コンテンツ鮮度アラート・季節提案機能の実装

### Phase 3：おしらせ配信・緊急モードの追加

**ゴール：** 生成したコンテンツを複数チャネルに一括配信でき、緊急時にも対応できる状態を作る。

- [ ] サイネージ端末への自動配信機能の実装
- [ ] メール・LINE一括配信機能の実装
- [ ] 配信スケジュール管理機能の実装
- [ ] 緊急モード（ワンタップ切り替え・緊急画面自動生成）の実装
- [ ] 配信履歴・確認状況の可視化機能の実装
- [ ] マンション管理士向け管理ダッシュボードの実装

## 8. 非機能要件

| 項目 | 方針 |
|------|------|
| UI・アクセシビリティ | 文字サイズ最小18px、ボタン最小44px以上、コントラスト比4.5:1以上 |
| 操作性 | 1つの目的につき3タップ以内で完結することを原則とする |
| セキュリティ | Supabase RLS（Row Level Security）による組合単位のデータ分離 |
| データ保管 | 議事録・お知らせは組合ごとに永続保管し、理事交代時も引き継ぎ可能 |
| 可用性 | 緊急モードは即時切り替え（5秒以内）を目標。Vercel / Supabaseのマネージドサービスで24時間365日稼働 |
| モバイル対応 | スマートフォン・タブレット対応（PCは補助的位置付け） |
| 著作権対応 | 生成画像・イラスト素材は商用利用可能なAPIのみ使用 |

## 9. 競合との差別化

| 競合 | 限界 | 番頭さんの優位性 |
|------|------|----------------|
| Mcloud・osumait | 管理会社・ITリテラシー高めユーザー前提の操作設計 | 「話しかけるだけ」で完結。番頭さんキャラでAIを隠した親しみやすいUX |
| 汎用AI議事録ツール（Notta等） | IT知識前提・管理組合業務に特化していない | 管理組合の業務フローに特化したスキル設計 |
| 既存サイネージCMS（OneGATE等） | 専任担当者が前提。AI生成機能なし・更新が止まる | 専任不要。番頭さんが先回りしてコンテンツ更新を促す |
| Canva等デザインツール | コンテンツ生成止まり。配信・スケジュール管理まで繋がらない | 生成から配信まで一気通貫 |
| 人手（管理士・管理会社） | コストが高い・対応時間に制約がある | 24時間いつでも・低コストで対応 |

## 10. 用語定義

| 用語 | 定義 |
|------|------|
| 番頭さん | 本プロダクトの名称。管理組合の情報を差配してくれる存在としてのコンセプト名 |
| 番頭さん | プロダクト内でAIを指す親しみやすい呼称。UIには「番頭さんが考えています…」のように使用 |
| はり出し | サイネージ・掲示板へのコンテンツ出力機能の名称（Phase 1コア機能） |
| 議事録まとめ | 音声録音から議事録を自動生成するスキルの名称（Phase 2） |
| おしらせ配信 | 複数チャネルへの一括配信スキルの名称（Phase 3） |
| スキル | 番頭さん内の個別機能モジュールの呼称 |
| ワンソース・マルチアウトプット | 1回の情報入力からサイネージ・PDF・テキスト等を同時生成すること |
| 賞味期限アラート | 掲示コンテンツが古くなった際に番頭さんが更新を促す通知機能 |
| 管理組合 | マンションの区分所有者で構成される自治組織 |
| 理事・理事長 | 管理組合の運営を担う役員。輪番制で毎年交代することが多い |
